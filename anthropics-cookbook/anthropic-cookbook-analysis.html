📚 anthropic-cookbook — Analyse

📖 Le récit

Comment Anthropic met en scène son propre savoir-faire.

1. Le décor

Imaginez une bibliothèque ouverte sur un campus. À l'entrée, une grande étagère plate : pas d'arborescence labyrinthique, pas de menus à cascade. Cinq rayonnages, cinq promesses — Capabilities, Tool Use, Multimodal, Third-Party, Advanced Techniques. Chaque rayonnage contient des fiches d'une seule page. Une fiche, une idée. Si vous cherchez "comment Claude classe des tickets", vous trouvez capabilities/classification/guide.ipynb — et rien d'autre. Pas de "v2 améliorée", pas de "guide avancé". La fiche, la vraie.

Le décor est volontairement plat. La profondeur vit dans le filesystem, pas dans la table des matières. C'est un parti pris pédagogique : l'utilisateur n'a pas à raisonner sur la hiérarchie pour trouver une recette.

2. Les règles du jeu

Trois règles tiennent l'édifice :

  • Un concept par notebook. Pas deux, pas trois. Si vous voulez expliquer le tool choice et les parallel tools, vous écrivez deux fichiers.
  • Le code reste pur, les métadonnées vivent ailleurs. Aucun YAML front-matter dans les notebooks. À la place, un registry.yaml central qui catalogue les 70 recettes avec leurs titres, descriptions, tags, auteurs.
  • Pas de modèles datés dans le code. claude-sonnet-4-6, claude-haiku-4-5 — des alias mouvants. La règle vise un objectif : que les recettes vieillissent gracieusement.
"Use environment variables for API keys" — règle non-négociable. Aucun secret ne franchit le seuil du commit.

3. Le socle immuable

I1 Tout exemple est un notebook exécutable. Pas de pseudo-code. Pas de tutoriel HTML.
I2 Les secrets passent par os.environ et un .env.example. Jamais dans le repo.
I3 Les modèles sont référencés par alias non-daté.
I4 Les métadonnées sont externes (registry.yaml, authors.yaml).
I5 Les commits suivent la convention feat/fix/docs/style/refactor/test/chore/ci.

4. Les leviers de transformation

Trois leviers font évoluer le cookbook sans le casser :

  • L'ajout de buckets dans registry.yaml — quand une nouvelle famille de patterns émerge (Skills Framework, Managed Agents), un bucket apparaît. Les anciens ne bougent pas.
  • La numérotation des séries pédagogiques — 00_The_one_liner_research_agent.ipynb à 06_The_vulnerability_detection_agent.ipynb. La progression est lisible d'un coup d'œil.
  • L'externalisation des prompts — dans patterns/agents/prompts/, les prompts sont des fichiers séparés, versionnés, diffables.

5. Les points de rupture

SeuilConséquence
Notebook qui couvre 2 concepts Refusé en review — règle "one concept per notebook"
Modèle daté hardcodé Casse la promesse de fraîcheur — alias requis
Secret commit Refusé — pre-commit hook + revue manuelle
Front-matter YAML dans un .ipynb Pollue le code, conflit avec registry.yaml
Notebook sans cellule markdown d'intro Échoue le critère pédagogique implicite

6. Le chemin de moindre résistance

README plat
→
Bucket (5 sections)
→
Notebook unique
→
Exécution end-to-end
→
Copier le pattern

Le happy path est court : un développeur arrive sur le README, repère sa famille de besoin (RAG, tool use, vision…), clique sur une entrée, ouvre le notebook, exécute. Pas d'étape de configuration transverse, pas de framework propriétaire à apprendre. Le coût d'entrée est presque nul.

7. L'équilibre du risque

Le cookbook est un actif documentaire à très faible risque de lock-in : code MIT, pas de SDK propriétaire imposé, notebooks ré-écrivables hors-Anthropic. Le risque réel se loge ailleurs — dans la dérive de fraîcheur : 70 recettes, nouveaux modèles trimestriels. La discipline des alias non-datés et du registry central est ce qui maintient la cohérence.

Pour un utilisateur qui veut cartographier l'architecture Anthropic, ce repo est la pierre de Rosette : sa taxonomie en 11 buckets, son découpage en 3 niveaux d'encapsulation (Skills → SDK → Managed), sa convention "un concept par fichier" forment un modèle reproductible sans frais.

🔬 Analyse fmaths (8 couches)

L0Ontologie

Primitives non-redondantes du cookbook :
  • Recette — unité atomique (un notebook).
  • Bucket — famille de recettes (Capabilities, Tool Use, etc.).
  • Registry — fichier de métadonnées central (registry.yaml).
  • Convention — règle de style/naming/commit (CLAUDE.md, CONTRIBUTING.md).
  • Niveau d'encapsulation — Skills, Agent SDK, Managed Agents.
  • Pattern — schéma agent réutilisable (Routing, Orchestrator-Workers…).
  • Asset — donnée d'appui (CSV, JSON, image, prompt).

L1Ensembles & relations

  • R = ensemble des recettes (≈ 70)
  • B = ensemble des buckets (11)
  • r ∈ R → b ∈ B : chaque recette appartient à exactement un bucket (partition).
  • Registry = fonction r → metadata(r).
  • Encapsulation = ordre partiel Skills ≼ SDK ≼ Managed (poids opérationnel croissant).

L2Logique & connecteurs

  • AND : un notebook valide ⟺ exécutable ET sans secrets ET alias non-daté ET single-concept.
  • XOR : un projet d'agent choisit Skills XOR SDK XOR Managed (les trois ne se mélangent pas dans le même runtime).
  • OR : les patterns d'agents (chaining / routing / parallel / orchestrator / evaluator) sont composables.
Tension : la promesse "self-contained notebook" s'oppose à la mutualisation de code (util.py, memory_tool.py). Le repo tranche au cas par cas — utilitaire local si court, fichier .py séparé si réutilisé.

L3Invariants

I1 Une recette = un notebook exécutable.
I2 Aucun secret committé. .env.example + os.environ.
I3 Alias modèles non-datés (claude-sonnet-4-6).
I4 Métadonnées hors-notebook (registry.yaml, authors.yaml).
I5 Conventional commits, branches <user>/<feat>.
I6 One concept per notebook.
I7 README plat (5 sections fixes).

⚙️ Installation

Le repo n'est pas une lib à installer stricto sensu — c'est une collection de notebooks à exécuter. Le setup vise donc à reproduire l'environnement.

Prérequis

OutilVersionVérification
Python>=3.11, <3.13python --version
uvrécentuv --version
Git2.xgit --version
Clé API Anthropicn/aconsole.anthropic.com

Option A — Lecture seule (sans clone)

Pour cataloguer les patterns (cas du skill architecture-anthropic), pas besoin de cloner. Lire directement via raw.githubusercontent.com/anthropics/anthropic-cookbook/main/....

Option B — Exécution locale (depuis la source)

# 1. Cloner le repo
git clone https://github.com/anthropics/anthropic-cookbook.git
cd anthropic-cookbook

# 2. Installer uv si absent
# Windows PowerShell :
powershell -ExecutionPolicy ByPass -c "irm https://astral.sh/uv/install.ps1 | iex"

# 3. Synchroniser les dépendances
uv sync

# 4. Copier le template d'env
cp .env.example .env

# 5. Éditer .env et y mettre ANTHROPIC_API_KEY=sk-ant-...

# 6. Lancer Jupyter
uv run jupyter lab

Config minimale (.env)

ANTHROPIC_API_KEY=sk-ant-xxxxxxxxxxxxxxxxxxxxx
# Pour Skills (beta) — uniquement si vous explorez /skills :
# ANTHROPIC_BETA="code-execution-2025-08-25,files-api-2025-04-14,skills-2025-10-02"

Vérification

uv run python -c "import anthropic; print(anthropic.__version__)"
# attendu : >= 0.77.0

uv run pytest tests/ -v
# attendu : tests verts (la suite est volontairement légère)

Points d'attention Windows

PowerShell vs bash. Les notebooks utilisent parfois !command dans les cellules : ces commandes POSIX échouent sur Windows. Préférer WSL2 ou uv run jupyter directement.
Encodage UTF-8. Les notebooks contiennent des caractères Unicode. Sur Windows, vérifier que la console est en UTF-8 (chcp 65001) avant d'exécuter des scripts liés.
Chemins longs. Activer LongPathsEnabled=1 dans le registre si vous clonez dans un chemin profond, sinon certains assets Skills peuvent poser problème.
Pour le skill architecture-anthropic : Option A suffit. Pas besoin d'installer l'environnement Python — on cartographie, on n'exécute pas.

Recommandation finale

1. Pourquoi maintenant. Pour un skill qui prétend cartographier "l'architecture Anthropic officielle", il n'existe pas de meilleure source primaire. La taxonomie en 11 buckets et la grille des 3 niveaux d'encapsulation (Skills / SDK / Managed) sont stables, lisibles, et explicitement promues par Anthropic dans son registry.yaml.

2. Action recommandée. Adopter registry.yaml comme nomenclature de référence dans architecture-anthropic. Reprendre les invariants I1–I7 comme checklist d'audit perso. Documenter explicitement la grille de décision Skills → SDK → Managed. Re-auditer ce repo tous les ~6 mois ou à chaque nouveau bucket dans registry.yaml.
