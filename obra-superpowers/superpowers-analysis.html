Le décor

Il y a une dizaine d'agents de code sur le marché, chacun avec ses conventions, ses formats, ses chemins d'invocation. Superpowers est né d'une idée simple — et un peu folle : écrire une méthodologie de développement une seule fois, et la faire fonctionner partout.

Le résultat, c'est un repo de 206 000 étoiles, 14 skills opérationnels, et cinq formats d'installation différents — un par agent. Claude Code, Cursor, Codex, Gemini, Factory Droid, Copilot : tous peuvent consommer les mêmes protocoles. Le décor, c'est une bibliothèque de comportements codés en markdown, sans framework, sans build step, sans dépendance runtime. Juste des instructions.

💬 Brainstorm→ 🌿 Worktree→ 📋 Plan→ ⚙️ Execute→ 🔴 TDD→ 🔍 Review→ ✅ Finish

Les règles du jeu

La règle fondamentale est contre-intuitive dans un monde qui valorise la vitesse : tu ne codes pas avant d'avoir un test qui échoue.

RED-GREEN-REFACTOR n'est pas une option. C'est un invariant. L'agent doit produire un test failing, puis le faire passer, puis nettoyer. Tout autre ordre est explicitement interdit par le skill test-driven-development.

La deuxième règle : une feature = un worktree = une branche. Pas d'entrelacement, pas de "j'ai presque fini, je rajoute juste ça". La branche est sacrée jusqu'à la review.

La troisième règle, ajoutée en v5.1 après une phase de friction : plus de dispatch nommé. Les agents ne s'appellent plus par nom propre. On passe par des templates génériques — sacrifice d'expressivité pour gain de portabilité.

"La discipline est le prix de la vitesse sur le long terme."

Le socle immuable

Superpowers repose sur des invariants que deux ans d'itération n'ont pas bougés : le workflow séquentiel en 7 étapes est intouchable. L'ordre n'a jamais été remis en cause, même lors des refactors majeurs. La vérification avant completion est le dernier filet. Le skill verification-before-completion est celui qui autorise le merge — sauter cette étape, c'est introduire un bug silencieux.

Avertissement structurel : les skills de superpowers n'ont pas de frontmatter YAML (contrairement à anthropics/skills). Aucune validation automatisée possible — un skill mal écrit passe inaperçu jusqu'à l'exécution.

Les leviers de transformation

Le premier levier : le parallélisme des subagents. Le skill dispatching-parallel-agents découpe un plan en N trajectoires simultanées. C'est là que le framework passe du "je code mieux" au "je code plus vite et mieux".

Le deuxième levier : l'extension est ouverte. Le skill writing-skills est dans le repo. La méthodologie se documente elle-même. N'importe quelle équipe peut créer ses propres skills et les distribuer via le même mécanisme de plugin.

Le troisième levier — souvent sous-estimé — est la portabilité multi-agent. Une équipe qui migre de Cursor vers Claude Code ne perd pas sa méthodologie. Elle change juste de manifest.

Risque insidieux : si les hooks ne chargent pas, l'agent continue de fonctionner — mais sans la méthodologie. Aucune erreur visible. Le résultat se dégrade silencieusement.

Le chemin de moindre résistance

Pour un projet neuf sur Claude Code :

/plugin install superpowers@claude-plugins-official

Puis la session démarre, les hooks s'initialisent, les 14 skills deviennent disponibles. L'agent propose /brainstorm. Le problème se précise en questions socratiques. Un plan en tâches 2-5 min est validé. Un worktree s'ouvre. Les subagents s'exécutent. Les tests rougissent avant de verdir.

C'est le chemin le plus court vers du code qui tient en production. Pas le chemin le plus immédiat — mais le moins cher sur la durée.

L'équilibre du risque

Le risque principal n'est pas la qualité du framework — elle est indéniable. C'est le rythme des breaking changes. Trente versions en huit mois. V4→V5 a cassé le dispatch nommé. V6 est probable en 2026.

Pour une adoption individuelle : risque faible. Pour une intégration dans un produit avec SLA : épingler la version et auditer chaque upgrade. Un breaking change non anticipé dans le manifest peut silencieusement désactiver toute la méthodologie.

Adoptez-le. Épinglez la version. Testez les upgrades en worktree isolé — le framework vous l'enseigne lui-même.

L0 Ontologie

  • Skill — protocole markdown auto-suffisant définissant un comportement agent (sans frontmatter YAML)
  • Agent — moteur d'exécution consommateur (Claude Code, Cursor, Codex, Gemini CLI, Factory Droid, Copilot)
  • Plugin — bundle d'installation multi-agents encapsulant N skills sous un format d'agent spécifique
  • Hook — déclencheur événementiel (session-start, pre/post-tool) orchestrant le bootstrapping du workflow
  • Plan — artefact structuré de tâches temporées (2-5 min chacune), produit par writing-plans
  • Worktree — espace de travail Git isolé sur branche dédiée, activé par using-git-worktrees
  • Subagent — processus délégué lancé par dispatching-parallel-agents pour exécution parallèle

L1 Ensembles et relations

  • Skills = 14 éléments (brainstorming, dispatching-parallel-agents, executing-plans, finishing-a-development-branch, receiving-code-review, requesting-code-review, subagent-driven-development, systematic-debugging, test-driven-development, using-git-worktrees, using-superpowers, verification-before-completion, writing-plans, writing-skills)
  • Agents_supportés = 6 éléments (Claude Code, Codex CLI, Factory Droid, Gemini CLI, Cursor, GitHub Copilot)
  • Manifests = 5 formats (.claude-plugin/, .codex-plugin/, .cursor-plugin/, .opencode/, gemini-extension.json)
  • Workflow_stages = 7 étapes ordonnées (Brainstorm → Worktree → Plan → Execute → TDD → Review → Finish)
  • Relations : encapsulates: Plugin → Skills (1:N) · targets: Plugin → Agent (1:1) · ordered: Workflow_stages

L2 Logique et tensions

AND : valid_delivery ⟺ brainstorm_done ∧ worktree_isolated ∧ plan_approved ∧ tdd_red_green_refactor ∧ code_review_passed ∧ verification_ok

XOR (historique) : named_dispatch(< v5.1) ⊕ generic_template_dispatch(≥ v5.1) — rupture backward-incompatible

OR : agent peut consommer skills en séquentiel OU via dispatching-parallel-agents

Tension #1 — Universalité vs Fidélité : 6 agents supportés → 5 formats de manifests distincts à maintenir en sync. Toute évolution d'un skill = 6 mises à jour potentielles.
Tension #2 — Légèreté vs Richesse : installation en 1 commande ↔ 14 skills + hooks + worktrees = complexité cachée post-install.
Tension #3 — Discipline vs Agilité : workflow 7 étapes ordonné ↔ tendance naturelle des agents à sauter des étapes sous pression.
Tension #4 — Agnosticisme vs Spécialisation : skills sans frontmatter YAML (agnostiques) ↔ manifests hyper-spécialisés par agent.

Vérification de l'installation

# Dans une nouvelle session Claude Code
/skills           # Liste les skills disponibles
/brainstorm       # Démarre une session socratique
/using-superpowers # Guide complet du workflow
Résultat attendu : Claude propose une session de brainstorming. Si les skills ne répondent pas, les hooks n'ont pas chargé — relancer la session.

Points d'attention Windows (PowerShell)

hooks-cursor.json : Cursor sur Windows utilise un fichier de hooks séparé (hooks/hooks-cursor.json). Si les skills ne s'activent pas dans Cursor, vérifier que ce fichier est bien chargé.
run-hook.cmd : Le fichier hooks/run-hook.cmd est le wrapper Windows pour les hooks shell. Il doit rester dans hooks/ — ne pas déplacer.
Paths avec espaces : Si votre dossier de projet contient des espaces, encapsuler les paths : "C:\Users\mon nom\projects\..."
Version épinglée : Superpowers sort ~30 releases en 8 mois. Spécifier la version à l'install : /plugin install superpowers@5.1.0

Recommandation finale

Superpowers est le framework de référence pour toute équipe qui utilise des agents de code. Rien d'autre sur le marché ne propose une méthodologie aussi complète, aussi battle-tested (206k ★, 2+ ans), aussi portable (6 agents). Le fait qu'il n'ait aucune dépendance runtime le rend quasiment sans risque à l'installation.

Action recommandée : adopter immédiatement pour tout nouveau projet avec agent de code. Épingler la version à 5.1.0. Créer un worktree dédié pour tester les upgrades avant de les pousser en production — le framework enseigne lui-même cette pratique. Pour les équipes sur Cursor Windows : valider manuellement le chargement de hooks-cursor.json après install. Pour les équipes qui veulent étendre : lire writing-skills en premier.
